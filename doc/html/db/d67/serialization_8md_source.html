1 # Writing data to binary formats
2 
3 It is often useful to be able to serialize a population into a binary format and re-construct a serialized population.
4 The namespace fwdpp::io provides the following functions for these operations:
5 
6 * fwdpp::io::serialize_population
7 * fwdpp::io::deserialize_population
8 
9 ## Required specializations
10 
11 **Note** This section is only required if you use functions in namespace fwdpp::io.
12 
13 The above functions are templates that require certain specializations to be visible at compile time in order to work
14 properly. These specializations are of function objects to serialize/deserialize mutations, gametes, and diploids.
15 Their names are:
16 
17 * fwdpp::io::serialize_mutation and fwdpp::io::deserialize_mutation
18 * fwdpp::io::serialize_gamete and fwdpp::io::deserialize_gamete
19 * fwdpp::io::serialize_diploid and fwdpp::io::deserialize_diploid
20 
21 The default definitions of these types have the following behaviors:
22 
23 * fwdpp::io::serialize_mutation and fwdpp::io::deserialize_mutation throw an exception. Therefore, you *must*
24  specialize them for your mutation type. The built-in mutation types already have such specializations, which you may
25  study if you develop your own mutation type. See fwdpp::io::serialize_mutation<fwdpp::popgenmut> and
26  fwdpp::io::deserialize_mutation<fwdpp::popgenmut> for examples.
27 * The gamete serialization objects assume the gamete type is fwdpp::gamete. If this is not true, code may fail to
28  compile and you need a specialization.
29 * The diploid serialization code only deals with the `first` and `second` data fields. If your diploid type requires
30  additional meta-data, then you need a specialization. See fwdpp::io::serialize_diploid<custom_diploid_testing_t> and
31  fwdpp::io::deserialize_diploid<custom_diploid_testing_t>, which are part of the fwdpp unit test suite.
32 
33 ## Low-level functions
34 
35 The functions fwdpp::io::serialize_population and fwdpp::io::deserialize_population discussed above are implemented in
36 terms of the following low-level functions:
37 
38 * fwdpp::io::write_mutations and fwdpp::io::read_mutations
39 * fwdpp::io::write_gametes and fwdpp::io::read_gametes
40 * fwdpp::io::write_diploids and fwdpp::io::read_diploids
41 
42 You may find these functions useful for more nuanced approaches to serialization.

44 ## Streams
45 
46 The stream template types all assume interfaces that are modeled after
47 [std::basic_iostream](http://en.cppreference.com/w/cpp/io/basic_iostream). The boost
48 [iostreams](http://www.boost.org/doc/libs/1_66_0/libs/iostreams/doc/index.html) library will satisfy this criterion.
49 
50 ## General guidelines
51 
52 Working with streams is a C++ topic rather than a fwdpp topic. However, keep the following in mind:
53 
54 * The [sstream](http://en.cppreference.com/w/cpp/header/sstream) header may be used to serialize in-memory.
55 * The objects in [sstream](http://en.cppreference.com/w/cpp/header/sstream) return a `std::string` via the `str()`
56  member function. Further, a `std::string`'s `c_str()` member function returns the underlying C-like `char *`.
57 * Most compression libraries ([zlib](http://zlib.net), [bzip2](http://bzip2.org)) expect a `char *`.
58 
59 Therefore, you may serialize in-memory and then write the results to a compressed file.
60 
61 ## Limitations
62 
63 ### Large populations
64 
65 When simulating very large populations, it is conceivalbe that you could run out of RAM if you attempt to serialize
66 in-memory. One current workaround is to use the low-level object writing types (fwdpp::io::serialize_mutation *et al.*)
67 to roll your own serialization code that flushes buffers more often. If you take a look at the current serialization
68 code, you'll see that doing this would not be a difficult task. Another workaround may be to simply serialize to an
69 uncompressed temporary file.
70 
71 Future library versions may provide built-in functions
72 to facilitate serialization in "chunks".
