21  /// \brief Apply a mutation scheme to add neutral mutations to a fwdpp::ts::table_collection.
22  ///
23  /// \param r fwdpp::GSLrng_t
24  /// \param make_mutation A mutation function. See below.
25  /// \param tables A fwdpp::ts::table_collection
26  /// \param samples A list of sample nodes corresponding to "currently-alive" nodes
27  /// \param mu Mutation rate (per gamete, per generation)
28  ///
29  /// \version 0.7.0 Added to library
30  /// \version 0.7.2 Return immediately if mutation rate is not > 0
31  ///
32  /// This function uses the edge_table to apply mutations to all parent/child connections.
33  /// The mutations should be neutral, although that requirement is not enforced. (It simply
34  /// makes little sense to apply selected mutations to the entire edge table post-hoc.)
35  ///
36  /// The result of this function is to populate \a tables.mutation_table with neutral variants.
37  ///
38  /// The parameter \a make_mutation is a function that must conform to
39  /// std::function<new_variant_record(double, double, fwdpp::uint_t)>. The three arguments
40  /// are interpreted as "left", "right", and "time". The function's return value
41  /// allows us to properly update the site and mutation tables in \a tables.
42  /// The new mutation must have a position on the half-open interval [left, right). The
43  /// "time" parameger represents the generation when the mutation arose and will be
44  /// uniformly assigned between the parental birth time and that of the child.
45  /// This mutation's origin time will be on the half-open interval (parental birth time,
46  /// child birth time].
47  ///
48  /// The preceding paragraph implies that \a make_mutation is repsonsible for required operations
49  /// related to mutation recycling, etc.. The lambda "neutral_variant_maker" in the example
50  /// program wfts.cc will probably be of help here.
51  ///
52  /// It is common that, at the end of a forward-time simulation, that not all trees are
53  /// completely coalesced. This happens because the distribution on the time to MRCA
54  /// has very high variance and, due to recombination, some fraction of trees will not be
55  /// coalesced after, say, 10N generations of evolving a single Wright-Fisher deme. Further,
56  /// the simlification algorithm will "push" the most ancient nodes on these "marginal forests"
57  /// forwards in time to the most recent ancestral node of each tree in the forest. Thus, naively
58  /// mutating the edge table will place too few mutations on these parts of the genome. This
59  /// function corrects for the presence of marginal forests, as described below.
60  ///
61  /// The \a samples list is used to identify "marginal forests", meaning marginal trees
62  /// that are not completely coalesced. (The finding is done via a call to
63  /// fwdpp::ts::mark_multiple_roots.) The MRCA nodes on trees in marginal forests have additional
64  /// mutations on them, representing evolution from time point zero (the beginning of the simulation)
65  /// to the MRCA node's time.
66  ///
67  /// Note that two alternatives exist that will render the treatment of marginal forests unnecessary:
68  /// 1. Simulate for longer. For example, CDF of TMRCA under Wright-Fisher is quite close to 1 at ~20N generations.
69  /// Thus, simulating longer means that fewer and fewer marginal trees will be forests.
70  /// 2. Start the simulation with an existing, completely-coalesced, tree sequence. As far as fwdpp is concerned,
71  /// this is left as an "exercise for the reader" at the moment.
72  {
