41 
42 * Added fwdpp::ts::generate_offspring, which is implemented in terms of fwdpp::mutate_recombine, and returns the
43 data structures needed for edge recording after creating an offspring individual. This new function streamlines
44 the API for "meiosis", with the same function applicable to fwdpp::slocuspop and fwdpp::mlocuspop objects.
45 * The constructor for fwdpp::poptypes::mlocuspop is changed to require the locus boundaries. **This change breaks
46  API**. The result is a much safer type.
47 * Added fwdpp::GSLrng_mt as a typedef for a mersenne twister
48 * Added fwdpp::zero_out_gametes, which is a convenience functions for a routine step needed each generation
49 * Added fwdpp::mendel, which is a convenience function for a routine step needed each generation
50 * Made updates to fwdpp::make_genetic_parameters, fixing the perfect forwarding and adding the ability to have
51 custom replacements for the functionality of fwdpp::mendel.
52 * The constructor of fwdpp::gamete_base taking an unsigned int is now marked explicit. (Having it non-expicit was a
53  bone-headed error that made an initial bug in fwdpp::zero_out_gametes very difficult to track down. See [this
54  commit](https://github.com/molpopgen/fwdpp/pull/178/commits/6af6a101db08a688ea030509c27f0acea41dab1f).
55 * Adds example program wfmlts.cc
56 * wfts.cc and wfmlts.cc now share a lot of common code, showing the power of a generic API.
57 * Fixed a bug in lambda captures affecting some test suite fixtures. [commit](https://github.com/molpopgen/fwdpp/pull/178/commits/7ace1fb9f4efa0c963c13150ddfd023786f31d28)
58 * Added a new test suite module that stress-tests fwdpp::ts::generate_offspring for the multi-locus case. The new test
59  suite fixtures are also used as more/better unit tests of fwdpp::fwdpp_internal::multilocus_rec_mut.

76 ## 0.7.3

78 * The return value of fwdpp::ts::table_simplifier::simplify has been changed to include a list of indexes to mutations
79  that are preserved by simplification. This return value may be passed to a new overload of fwdpp::ts::make_mut_queue

139 ## 0.6.0
140 
141 This is a major release, breaking API compatibility in many areas. These changes were necessary to make the library
142 even more flexible, but also to undo several poor design choices that we've been living with for far too long. This
143 release can be seen as a big push towards a 1.0 release.
144 
145 The result of these changes is a much smaller library that is safer and can do more.
146 
147 The following bug fixes are included in this release. None affected
148 [fwdpy11's](http://fwdpy11.readthedocs.io/en/latest/) official releases, either because that
149 package uses custom replacements of fwdpp functions and/or they were caught before a release was made or are in features
150 so new that no one has affected by them:
151 
152 * A bug in which selected fixations were not recorded by fwdpp::update_mutations_n was fixed.
153  [PR119](https://github.com/molpopgen/fwdpp/pull/119)

355 ## 0.4.4
356 
357 TL;DR Big performance improvements due to better handling of objects in memory. Yay! API changes. Boo!
358 
359 __The library documentation is likely to be out of date until a future release__
360 
361 * Vastly improved management of object lifetimes:
362  * Extinct mutations/gametes are no longer removed each generation.
363  * FIFO queues are constructed each generation in order to "recycle" those objects as new mutations/gametes.
364  * The queue is implemented as std::queue<mlist::iterator> or std::queue<glist::iterator> for mutations and gametes, respectively.
365  * These queues result in big performance improvements for "big" simulations, at the cost of breaking API compatibility. The performance improvement is both reduced run time and reduced memory usage.
366  * In order to accomodate the "recycling", mutation models must now return iterators derived from the mutation list, rather than mutation types themselves.
367  * Additionally, the internal mutation/recombination functions must take non-const references to "recycling bins", which are the FIFO queues.
368  * Mutation type data members (objects inheriting from KTfwd::mutation_base) are no longer const. This is required in order to enable the "recycling".
369  * In order to take advantage of this feature, extinct mutations must not be removed each generation. The function KTfwd::update_mutations should be removed instead of KTfwd::remove_lost, KTfwd::remove_fixed_lost, etc. The latter functions are marked as deprecated.
370  * The details of recycling are handled by KTfwd::fwdpp_internal::make_mut_queue, KTfwd::fwdpp_internal::make_gamete_queue, KTfwd::fwdpp_internal::recycle_gamete, and the very cool variadic template function KTfwd::fwdpp_internal::recycle_mutation_helper. These types/functions are found in fwdpp/internal/recycling.hpp.
371 
372 * Improved memory managament:
373  * "Recycling" means that the linked lists used are much more constant in memory with respect to allocations. Thus, it makes sense to allocate objects prior to simulation.
374  * The functions KTfwd::add_elements in the main library and KTfwd::add_recyclable in the sugar sub-library allow the addition of recylable objects. See example programs for how to use these functions.
375 
376 * Other API changes:
377  * As a result of the changes described above, the data structures of the library were changed from linked lists of iterators to vectors of std::size_t. This is a radical change to the data layout, and results in further speedups.
378  * KTfwd::sample_diploid has been streamlined in light of common defaults
379  * Recombination policies have been simplified. See KTfwd::poisson_xover for an example.
380 
381 * Behavior changes:
382  * As a result of object recycling, data structures (mutation and gamete lists, specifically) at the end of a simulation contain both extant and extinct objects.
383 * Implementation changes:
384  * Serialized populations may contain extinct mutations and gametes.
385  * A lot of redundant code has been replaced with function calls. This should help prevent bugs like Issue #27, which was due to a botched copy-paste, which happens when code is written in a hurry...
386  * Simpler dispatch method for mutation models (KTfwd::fwdpp_internal::mutation_model_dispatcher)

388 * Improved "type traits" sub-library (namespace KTfwd::traits)
389 * Deprecated functions/objects removed from library
390 * Bug fixes:
391  * Issue #30 fixed regarding serialization of KTfwd::generalmut and KTfwd::generalmut_vec
392  * A bug in KTfwd::ms_sample and KTfwd::ms_sample separate was identified and fixed. The bug only applied to multilocus simulations. The bug was that the first 'n' chromosomes were sampled, rather than 'n' randomly-chosen chromosomes. The effect of the bug is that the first sample is truly random (as chromosomes are not sorted in any meaningful way), but a second sample would overlap with the first, etc.

469 ##0.3.3
470 
471 This release of __fwdpp__ includes major performance improvements. The short version of the story is:
472 
473 * Linked lists are traversed less often each generation compared to previous versions.
474 * An (extremely) expensive linear-time search of a linked list that previously occurred after _every_ recombination event has been eliminated, and replaced with a very fast log-time search of a lookup table.
475 * The containers required for storing the intermediate steps of recombination are now allocated at most once per replicate, and their capacity is adjusted as needed during the simulation. Previously, these containers were allocated for every crossover event.
476 * The way that mutations are copied during recombination has been changed from a call to std::copy to the vector's insert member function, resulting in fewer reallocations during copying.
477 
478 The result is much better scaling with large population size and recombination rates.
479 
480 Major changes:
481 
482 * Crossing over has been streamlined. Unfortunately, this changes the library API. However, run times improve substantially.
483 * The library internals now use different insertion methods during recombination. We swtiched from copy(beg,end,back_inserter(x)) to x.insert(x.end(),beg,end), which results in less memory usage, and some run-time improvement for large simulations.
484 * API change: the simplification of metapopulation containers in 0.3.2 means that we can make the recombination policies required for such simulations the same as for single-deme simulations.
485 * The build system has been streamlined, with fewer dependencies and a greater emphasis on standard containers + Google's perftools.
486 * The various read/write functions for serializations have been renamed--I simply couldn't get the template declarations to auto-deduce the types. I gave up on this one, and so there's an API change. On the plus side, IO.hpp/IO.tcc are more readable now.
487 
488 Minor changes:
489 
490 * KTfwd::serialize now has a default constructor and a move constructor defined, allowing it to be a member of another class, which helps things in [foRward](http://github.com/molpopgen/foRward).
491 * KTfwd::serialize can now be used to serialize multiple records into a single buffer.
492 * A "validation suite" has been added. This is mostly for the developer to have an automatic way to check for problems.
493 * The package now installs a single binary called fwdppConfig. This program is intended to be used in configure scripts to make checking for fwdpp's existence and/or version easier. It takes a single option:

495 ~~~{sh}
