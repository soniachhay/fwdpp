13 \section TutGeneral Policies in C++
14 
15 The behavior of a function may be modified by a policy. For example, [std::sort] defaults to sorting in ascending
16 order:
17 
18 ~~~{.cpp}
19 std::vector<int> x{5,4,3,2,1};
20 std::sort(x.begin(),x.end()); //x will now contain 1,2,3,4,5
21 ~~~
22 
23 This default behavior may be modified by passing a custom function ("policy") as an additional argument. In C++, this
24 function can take on several different forms, including function objects, lambda expression, and instances of
25 std::function. Here is an _incomplete_ list of the various ways you could sort in ascending order:
26 
27 ~~~{.cpp}
28 #include <algorithm>
29 #include <vector>
30 #include <functional>
31 
32 using namespace std;
33 
34 int main(int argc, char ** argv) {
35  vector<int> x{5,4,3,2,1};
36 
37  sort(x.begin(),x.end()); //x is now 1,2,3,4,5
38 
39  //sort x in ascending order via std::greater...
40  sort(x.begin(),x.end(),bind(greater<int>(),placeholders::_1,placeholders::_2));
41 
42  //...or a lambda function...
43  sort(x.begin(),x.end(),[](int a,int b) { return a>b; });
44 
45  //...or a std::function wrapping a lambda...
46  std::function<bool(int,int)> sorter = [](int a,int b){return a>b;};
47  sort(x.begin(),x.end(),sorter);
48 
49  //...or a std::function wrapping bind/greater...
50  sorter = std::bind(greater<int>(),placeholders::_1,placeholders::_2);
51  sort(x.begin(),x.end(),sorter);
52 }
53 ~~~

55 \section fwdppPolicies Policies in fwdpp
56 
57 Just like the STL, custom functions modify the behavior of simulations. These policies affect how mutation,
58 recombination, and selection all work. Additionally, policies may affect how mutations are removed from gametes and/or
59 the entire population at the end of each generation. For example, consider a neutral mutation that fixed in the most
60 recent generation. There is really no point in keeping a key to it in every gamete. Rather, we may prefer to remove it
61 from each gamete and store it in a vector of fixations. Doing so requires a policy passed to fwdpp::sample_diploid
62 telling it to remove neutral mutations. In the case of selected mutations, we may or may not want to remove such fixed
63 variants from gametes, depending on the context of our simulation (standard pop-gen where fixed selective mutations no
64 longer affect relative fitness vs quantitative trait simulations where they do contribute to trait values, and thus
65 affect distance to optimium, etc.).

67 \subsection TutMut Mutation policies
68 
69 A mutation model has one of the following function signatures. Either
70 
71 ~~~{.cpp}
72 std::size_t mutmodel(queue_t &,mcont_t &);
73 ~~~
74 
75 or
76 
77 ~~~{.cpp}
78 std::size_t mutmodel(gamete_t &,mcont_t &, queue_t &);
79 ~~~
80 
81 The former is used when the current state of a gamete doesn't matter. The infinitely-many sites model is an example.
82 The latter is used whn the current state of a gamete does matter. A finite-sites model would be an example.
83 Internally, the library detects which type is being used. The return value is the index in the mutation container where
84 the new mutation was placed. This location may have been generated via recycling, and the library provides
85 fwdpp::fwdpp_internal::recycle_mutation_helper to facilitate this operation.
86 
87 Additional model parameters are possible via the usual mechanisms: class members, std::bind, etc.
88 
89 A valid mutation policy passes a static assertion involving fwdpp::traits::valid_mutation_model at compile time. See
90 type_traitsTest.cc for an example of this assertion.
91 
92 For an example of how to compose a mutation policy, see @ref custom_mutation.cc.
